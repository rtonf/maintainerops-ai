## Threat Model

The main security boundary is public distribution:

- GitHub Marketplace users may copy the README workflow or install the Action by tag.
- npm users install the CLI package from the registry.
- Reviewers inspect public security/application evidence before trusting the project.
- Untrusted pull request and issue content must remain data, not executable instructions.
- Generated build artifacts should not be required in GitHub unless explicitly intended.

## Findings

No reportable security findings were introduced by the `v0.1.3` cleanup.

## Validation Notes

- The Action remains human-in-the-loop and read-only in the documented workflow.
- The README and Marketplace examples now point to `rtonf/maintainerops-ai@v0.1.3`.
- The Action wrapper only forwards `--offline` when `INPUT_OFFLINE=true`.
- The Action wrapper only forwards `--authorized` when `INPUT_AUTHORIZED=true` or `MAINTAINEROPS_AUTHORIZED=true`.
- npm package contents were reduced from 56 files to 32 files in dry-run output.
- Generated test files, eval runner output, and source maps are no longer included in the npm package dry run.
- Removed public files were internal design/draft artifacts and did not contain runtime code, credentials, or security controls.
- No live OpenAI, npm, GitHub, or private key patterns were found in publishable files.

## Residual Risk

The composite Action runs `npm ci` and `npm run build:cli` from the tagged Action source. This avoids committing generated `dist/` to GitHub, but consumers should pin a release tag such as `v0.1.3` rather than an untrusted moving branch.

External maintainer feedback remains the main non-security evidence gap.
